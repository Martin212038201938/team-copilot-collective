## C2 — Cache-Control für Assets

### Befund: bereits umgesetzt (seit 03.02.2026)

`public/.htaccess` enthält seit Commit `bfcff62` ("SEO-Fix: .htaccess optimiert, neuer Artikel, llm.txt", 2026-02-03) einen vollständigen `CACHE-CONTROL`-Block:

```apache
<IfModule mod_headers.c>
    # Öffentliche Bilder (ungehashed): 30 Tage
    <FilesMatch "\.(png|jpg|jpeg|gif|webp|avif|svg)$">
        Header set Cache-Control "max-age=2592000, must-revalidate"
    </FilesMatch>

    # Gehashte Vite-Assets (z.B. index-Cxjk9haj.js): 1 Jahr + immutable
    <FilesMatch "^.+-[A-Za-z0-9]{8}\.(js|css|woff|woff2|ttf|eot|otf|png|jpg|jpeg|gif|webp|avif|svg)$">
        Header set Cache-Control "max-age=31536000, immutable"
    </FilesMatch>

    # HTML (react-snap pre-gerendert): kein stale Content, aber schnelle 304er
    <FilesMatch "\.html$">
        Header set Cache-Control "no-cache"
    </FilesMatch>

    # SEO-Dateien (txt/xml): 1 Stunde
    <FilesMatch "\.(txt|xml)$">
        Header set Cache-Control "max-age=3600, must-revalidate"
    </FilesMatch>

    # Favicon: 1 Tag
    <FilesMatch "\.(ico)$">
        Header set Cache-Control "max-age=86400, must-revalidate"
    </FilesMatch>
</IfModule>
```

Der Block ist auch im Build-Output (`dist/.htaccess`) vorhanden und wurde in diesem Lauf **live verifiziert**:

```
curl -sI https://copilotenschule.de/           → cache-control: no-cache
curl -sI https://copilotenschule.de/assets/index-Q0BhfIBu.js → cache-control: max-age=31536000, immutable
```

Beide Werte entsprechen der Konfiguration exakt. Die Strategie ist sinnvoll gestuft (HTML immer frisch validieren, gehashte Assets aggressiv cachen, ungehashte Bilder moderat) und braucht **keine weitere Änderung**.

### Warum stand C2 trotzdem noch als "offen" im Plan?

Die Maßnahme wurde am 27.05. bei der Plan-Neuaufnahme pauschal als 🔵 offen registriert, ohne gegen den tatsächlichen `.htaccess`-Stand zu prüfen (der Fix lag zu dem Zeitpunkt schon 3+ Monate zurück, 03.02.). Das ist ein Dokumentations-Nachzügler, keine echte Lücke.

### Empfehlung

## C1 — PageSpeed-API-Quota lösen

### Befund

Die `website-health-check`-Skill-Dokumentation (`SKILL.md`, "Modul 1: Technische Performance") sieht vor, für bis zu 10 Unterseiten pro Domain die **öffentliche PageSpeed-Insights-API v5 ohne API-Key** aufzurufen:

```
https://www.googleapis.com/pagespeedonline/v5/runPagespeed?url=...&strategy=mobile&category=performance&category=accessibility&category=best-practices&category=seo
```

Google begrenzt anonyme (keyless) Requests auf ein sehr niedriges, projektübergreifend geteiltes Kontingent. In den historischen Reports zeigt sich das wiederholt:

- `reports/2026-06-23.md`: *"PageSpeed API: Tagesquota überschritten — kein Core-Web-Vitals-Score heute"*
- `reports/2026-06-30.md`: *"PageSpeed: Quota überschritten (kein frischer Score)"*

**Wichtiger Nebenbefund:** Seit der Überarbeitung des täglichen Prompts am 27.05. (`SCHEDULED-TASK-PROMPT.md`) enthält der tägliche Workflow **kein PageSpeed-Modul mehr** — die Schritte a–j decken nur noch Verfügbarkeit/TTFB, GSC-Indexierung/-Leistung, Bing, Pre-Render/SSR, On-Page-Stichprobe, Wettbewerber, LLM-Sichtbarkeit und Protected Pages ab. Alle sieben geprüften Juli-Reports (01.–07.07.) enthalten **keine** PageSpeed-/Core-Web-Vitals-Erwähnung mehr.

**Konsequenz:** Das Quota-Problem wurde nicht gelöst, sondern durch Weglassen des Moduls umgangen. Damit fehlen aktuell dauerhaft Core-Web-Vitals-, Accessibility- und technische-SEO-Scores im Dashboard — ein stiller Funktionsverlust, kein aktiver Fehler mehr, aber auch keine echte Lösung. `SKILL.md` und `config.json` (`pagespeed_strategy: "mobile"`) sind dadurch inkonsistent mit dem tatsächlich gelebten Workflow.

### Lösungsoptionen

**Option A — Eigener Google-API-Key (empfohlen als Basis)**
- Kostenloser Key über Google Cloud Console (PageSpeed Insights API v5 aktivieren). Dokumentiertes Free-Quota: 25.000 Requests/Tag pro Key, statt eines sehr kleinen, global geteilten anonymen Kontingents.
- Ablage nach bestehendem Muster: `PAGESPEED_API_KEY` in `~/Documents/Cowork Bereich/website-health-check/.env` (nicht versioniert, analog zu `CLARITY_API_TOKEN`), Platzhalter in `.env.example` ergänzen.
- Aufruf-URL um `&key=$PAGESPEED_API_KEY` erweitern; Fallback auf keyless, falls kein Key gesetzt ist.
- **Kein Secret wird in versionierte Dateien geschrieben** — Key existiert nur lokal in `.env`, die bereits in `.gitignore` steht.

**Option B — Frequenz reduzieren + Caching**
- Statt täglich für bis zu 10 Seiten pro Domain: PageSpeed nur **1×/Woche** (z. B. montags) für **3 Kernseiten pro Domain** (Home, wichtigste `/wissen/`-Seite nach Klicks, wichtigste Trainings-/Workshop-Seite) abfragen.
- Ergebnis in `data/<domain>/pagespeed-cache.json` ablegen; an den übrigen Wochentagen den letzten Cache-Wert im Dashboard mit Datumsstempel weiterverwenden statt "kein Score".
- Reduziert Call-Volumen um ca. 90 % gegenüber der ursprünglichen Spezifikation.

**Empfehlung:** Option A + B kombinieren — eigener Key hebt die harte Quota-Grenze an, reduzierte Frequenz/Seitenzahl hält den Verbrauch zusätzlich niedrig und macht das System robust auch ohne Key. Das bringt Core-Web-Vitals-/Accessibility-/SEO-Scores zurück ins Dashboard, ohne das Quota-Risiko erneut einzugehen.

### Nächste Schritte (erfordern User-Handeln, außerhalb dieses Repos)

1. Google-Cloud-Projekt + PageSpeed-Insights-API-Key erstellen (kostenlos, ~5 Min.).
2. Key lokal in `~/Documents/Cowork Bereich/website-health-check/.env` als `PAGESPEED_API_KEY` eintragen.
3. `SKILL.md` (Modul 1) und `SCHEDULED-TASK-PROMPT.md` um die reduzierte Frequenz/Seitenzahl + Key-Nutzung ergänzen (liegt außerhalb von `team-copilot-collective`, daher hier nur als Empfehlung dokumentiert, kein Edit durch diesen Lauf).
4. Kein Deploy dieses Repos nötig — C1 betrifft ausschließlich das Health-Check-Tooling, nicht `copilotenschule.de` selbst.
